# 本地小模型实验（2026-06-10）

> 目标：验证给翻箱内置本地小模型做文件索引（截图打标 / md 语义关联 / 喂 agent）是否可行。
> **状态：已收尾。VLM 打标判定不可用；语义关联与 recall 验证可用（暂不集成）。**

## 结论（花叔拍板：VLM 太慢，不用了）

| 实验 | 结果 | 数据 |
|---|---|---|
| 实验1 截图打标（qwen3.5:0.8b） | ❌ **不可用** | 20张真实截屏：成功率仅 13/20（7张超时），均耗时 66.5s/张（含重试），顺利时也要 11-19s，1000 张估算 18.5 小时。质量本身合格（描述准、能抄出画面文字），死在速度和稳定性 |
| 实验2 md 语义关联（qwen3-embedding:0.6b） | ✅ 可用 | 181ms/篇，跨项目关联质量高（CC评测↔源码解构↔Boris技巧自动聚类），1000 篇约 3 分钟 |
| 实验3 fanbox-recall 喂 agent | ✅ 可用 | 独立 agent 实测：语义查询一次命中；暴露并修复了中文分词、静默吞错等 5 个问题 |
| 调研 3B 内 VLM Top10 | 见 `results/调研-3B内多模态模型Top10.md` | 0.8b 视觉能力垫底；慢的主因疑似 Ollama 视觉 prefill 路径而非模型大小 |

## 如果将来重启 VLM 方向，先看这三条

1. 换运行时再换模型：MLX-VLM / Moondream Station（M1 Max 实测 35tok/s）绕开 Ollama 视觉路径，比换小模型收益大。
2. arXiv 2601.19139：vllm-mlx 在 M4 Max 上图像缓存命中后单图 0.78s——「索引一次、缓存复用」的架构下延迟问题可能消失。
3. 送图分辨率降到 768px 以下再测，1024px 的视觉 token 是大头。

## 环境坑（重要，集成时必看）

- 系统代理（7890）会劫持 Ollama 出网导致 pull EOF 失败，registry 直连反而通。本次用 `env -u http_proxy …` 起的裸 server。**Ollama.app 常驻的话 pull 会持续失败。**
- Ollama 首次加载模型的窗口期，并发请求会 5 分钟超时 500；调用方必须带超时+重试+预热。
- node fetch 不设 timeout 会在请求被丢时永远挂死。
